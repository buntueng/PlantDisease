============================================================
1. WORKING TITLE
============================================================

Cross-Dataset Generalization of Lightweight Deep Learning for Multi-Class
Chilli Leaf Disease Classification


============================================================
2. CENTRAL RESEARCH IDEA
============================================================

The main scientific problem is not simply to obtain high accuracy on one
chilli leaf image dataset.

The study investigates whether lightweight deep learning models generalize
across independently collected public chilli leaf datasets that differ in
acquisition conditions, background, camera, geography, class distribution,
image quality, and dataset-specific characteristics.

The central motivation is:

    High within-dataset performance does not necessarily imply
    cross-dataset robustness.

Many plant-disease studies use random train/test splits from a single dataset.
Such evaluation can overestimate practical robustness because training and
test images may share the same visual style, acquisition process, background,
camera characteristics, and data-collection biases.

This work therefore emphasizes:

1. within-dataset evaluation;
2. bidirectional pairwise cross-dataset evaluation;
3. multi-source training with a completely held-out target dataset;
4. pooled multi-source evaluation as a secondary analysis;
5. efficiency and model-size comparison;
6. statistical significance and effect-size analysis.

The paper should be framed primarily as a cross-dataset generalization study,
not merely as another chilli-disease classification paper.


============================================================
3. MAIN RESEARCH QUESTIONS
============================================================

RQ1:
How much performance drops when a model trained on one public chilli dataset
is evaluated on another independently collected dataset?

RQ2:
Are lightweight architectures more or less robust to dataset shift than
larger CNN and Transformer baselines?

RQ3:
Does multi-source training improve performance on a held-out target dataset?

RQ4:
Can a compact proposed model provide a favorable trade-off among:
- cross-dataset Macro-F1;
- MCC;
- model parameter count;
- computational cost;
- memory use?

RQ5:
Are performance differences statistically significant and practically
meaningful?

RQ6:
Is a model with slightly lower within-domain accuracy potentially more useful
if it has a smaller cross-dataset generalization gap?

============================================================
7. DATA QUALITY AUDIT AND LEAKAGE CONTROL
============================================================

The fixed data pipeline applies the following conservative process:

1. recursively scan candidate image files;
2. exclude repository-provided augmented paths;
3. apply dataset-specific include/exclude path rules;
4. reject zero-byte files;
5. verify image readability with PIL;
6. reject corrupt or unreadable files;
7. infer canonical labels;
8. compute SHA-256 hashes for valid images;
9. detect exact duplicate image content;
10. detect exact duplicates assigned to conflicting labels;
11. exclude the entire conflicting hash group;
12. reduce same-label exact duplicates within a dataset to one deterministic
    representative;
13. preserve exact duplicates across datasets as a shared group ID for pooled
    group-aware CV;
14. remove exact source-target overlap before external evaluation.

Observed audit result:
- 14 files from 5 exact-duplicate SHA-256 groups were excluded because
  identical image content was associated with conflicting canonical labels.

Important policy:
- The code does not guess which conflicting label is correct.
- It does not use majority vote.
- It excludes the complete conflicting hash group conservatively.

Current verified external dry runs reported:

    exact_target_overlap_removed = 0

for all tested pairwise and multi-source scenarios.

Important limitation:
- SHA-256 only detects byte-identical duplicates.
- It does not detect:
  - resized copies;
  - cropped copies;
  - recompressed JPEG copies;
  - color-adjusted copies;
  - near-duplicates.
- A future near-duplicate audit using perceptual hashes or image embeddings
  would strengthen the study.

Audit outputs include:

data/manifests/
- training_index.csv
- scan_log.csv
- data_quality_summary.csv
- conflicting_duplicate_labels.csv
- removed_exact_duplicates.csv
- dataset_class_counts.csv
- dataset_class_matrix.csv


============================================================
8. EXPERIMENTAL DESIGN OVERVIEW
============================================================

Four main experiment families are implemented:

1. within_cv
2. pairwise
3. multisource
4. pooled_cv

These experiments must not be interpreted as equivalent.

------------------------------------------------------------
8.1 Experiment Family 1: Within-Dataset 5-Fold CV
------------------------------------------------------------

Datasets evaluated separately:

A -> A
B -> B
C -> C

Protocol:
- group-aware outer 5-fold CV;
- inner validation split from the development portion;
- early stopping uses validation Macro-F1;
- outer test fold is not used for early stopping or model selection.

Purpose:
- estimate in-domain performance;
- establish conventional baseline performance;
- quantify within-domain variance.

This is not external validation.

------------------------------------------------------------
8.2 Experiment Family 2: Bidirectional Pairwise Cross-Dataset
------------------------------------------------------------

For source S and target T, the active class space is:

Y_(S->T) = Y_S intersection Y_T

Confirmed dry-run structure:

A -> B:
5 classes
- bacterial_spot
- cercospora_leaf_spot
- curl_virus
- healthy
- nutrient_deficiency

B -> A:
same 5 classes

A -> C:
3 classes
- cercospora_leaf_spot
- healthy
- nutrient_deficiency

C -> A:
same 3 classes

B -> C:
4 classes
- cercospora_leaf_spot
- healthy
- nutrient_deficiency
- powdery_mildew

C -> B:
same 4 classes

Compact notation:

A^(5) -> B^(5)
B^(5) -> A^(5)

A^(3) -> C^(3)
C^(3) -> A^(3)

B^(4) -> C^(4)
C^(4) -> B^(4)

Purpose:
- directly test unseen-dataset generalization;
- test directionality/asymmetry of domain shift;
- avoid pretending absent classes exist.

Critical rule:
- target data are not used for source-model hyperparameter tuning;
- target data are not used for early stopping;
- model selection is source-side only;
- then the selected/frozen model is evaluated on the target.

------------------------------------------------------------
8.3 Experiment Family 3: Multi-Source Held-Out Target
------------------------------------------------------------

For source datasets S1...Sm and target T:

Y_((S1+...+Sm)->T)
=
(union of source labels) intersection target labels

Confirmed dry-run structure:

A + C -> B:
6 classes
- bacterial_spot
- cercospora_leaf_spot
- curl_virus
- healthy
- nutrient_deficiency
- powdery_mildew

A + B -> C:
4 classes
- cercospora_leaf_spot
- healthy
- nutrient_deficiency
- powdery_mildew

B + C -> A:
5 classes
- bacterial_spot
- cercospora_leaf_spot
- curl_virus
- healthy
- nutrient_deficiency

The headline multi-source experiment is:

    A + C -> B

because it evaluates all six harmonized classes on a fully held-out target
dataset B.

Purpose:
- determine whether source-domain diversity improves generalization;
- test a more realistic multi-source external evaluation.

Important limitation: class-source confounding.

For A+C -> B:

Class                     Source A   Source C
------------------------------------------------
bacterial_spot            yes        no
cercospora_leaf_spot      yes        yes
curl_virus                yes        no
healthy                   yes        yes
nutrient_deficiency       yes        yes
powdery_mildew            no         yes

Therefore:
- bacterial_spot source examples come only from A;
- curl_virus source examples come only from A;
- powdery_mildew source examples come only from C.

This does not invalidate held-out testing on B, but it means source-domain
identity and class identity are partially confounded for some classes.

The manuscript should describe this honestly.

------------------------------------------------------------
8.4 Experiment Family 4: Pooled 5-Fold CV
------------------------------------------------------------

Data:

A + B + C

Protocol:
- pooled group-aware 5-fold CV.

Purpose:
- secondary combined-data analysis;
- evaluate performance when all source domains are mixed.

Important:
- pooled CV is NOT external cross-dataset validation.
- It should be labeled as pooled multi-source classification or pooled CV.

10. PROPOSED LIGHTWEIGHT MODEL
============================================================

Working model name:

    ChilliLite-GFNet

Code name in configuration:

    chilli_lite_gfnet

Defined in:

    proposed_model.py

The proposed architecture is intentionally isolated from the baseline code so
it can be modified repeatedly without changing the common training protocol.

Current prototype design:

1. MobileNetV3-Small convolutional feature extractor;
2. three multi-scale feature taps;
3. lightweight 1x1 projections;
4. ECA-style channel recalibration;
5. spatial alignment of feature maps;
6. descriptor concatenation;
7. adaptive softmax gates across scales;
8. weighted multi-scale fusion;
9. depthwise-separable refinement;
10. global average pooling;
11. dropout;
12. final classifier.

Current feature-tap configuration:

tap_indices:
- 3
- 8
- 12

tap_channels:
- 24
- 48
- 576

projection_channels:
- 96

fusion_channels:
- 160

gate_hidden_channels:
- 64

dropout:
- 0.20

eca_kernel_size:
- 3

Conceptual equations:

For feature tap i:

B_i = ECA_i(
        activation(
          BN(
            Conv1x1_i(F_i)
          )
        )
      )

Descriptors:

z = concat(
      GAP(B_1),
      GAP(B_2),
      GAP(B_3)
    )

Adaptive branch gates:

alpha = softmax(
          W_2 * sigma(W_1 * z)
        )

with:

sum_i alpha_i = 1

Fused representation:

F_fuse = sum_i alpha_i * aligned(B_i)

Then depthwise-separable refinement and classification are applied.

Important:
- The current architecture is a research prototype.
- Do not automatically claim novelty only because components are combined.
- Novelty must be supported by literature comparison and ablation studies.
- The final proposed architecture may change.

1. Macro-F1

- class distributions differ;
- cross-dataset tasks use different class subsets;

============================================================
18. STATISTICAL ANALYSIS
============================================================

Implemented script:

    statistical_analysis.py

Planned evidence chain:

1. descriptive statistics;
2. Friedman omnibus test;
3. pairwise Wilcoxon signed-rank post-hoc tests;
4. Holm correction for multiple comparisons;
5. rank-biserial effect size;
6. exact McNemar test for proposed vs strongest baseline;
7. paired bootstrap 95% confidence interval for metric differences.

Significance level:

    alpha = 0.05

------------------------------------------------------------
18.1 Matched statistical units
------------------------------------------------------------

Within-CV / pooled-CV:
- matched by scenario + fold.

Pairwise / multi-source:
- matched by scenario + seed.

This prevents arbitrary comparison of unrelated runs.

------------------------------------------------------------
18.2 Friedman test
------------------------------------------------------------

Purpose:
- omnibus comparison across 3 or more models on matched units.

Null hypothesis:
- models have equivalent performance ranks.

------------------------------------------------------------
18.3 Pairwise Wilcoxon signed-rank
------------------------------------------------------------

Purpose:
- paired non-parametric comparison between models.

Examples:
- Proposed vs MobileNetV2
- Proposed vs MobileNetV3-Small
- Proposed vs EfficientNet-B0
- Proposed vs ResNet50
- etc.

------------------------------------------------------------
18.4 Holm correction
------------------------------------------------------------

Purpose:
- control family-wise error across multiple pairwise comparisons.

Primary inferential claims should use:

    Holm-adjusted p-values

not only raw p-values.

------------------------------------------------------------
18.5 Rank-biserial effect size
------------------------------------------------------------

Purpose:
- quantify practical magnitude of paired differences.

Do not rely on p-values alone.

------------------------------------------------------------
18.6 Exact McNemar test
------------------------------------------------------------

Used for proposed model vs strongest baseline on the same test images.

It analyzes discordant correctness:

- proposed correct, baseline wrong;
- proposed wrong, baseline correct.

This is especially appropriate because predictions are paired image-by-image.

------------------------------------------------------------
18.7 Paired bootstrap
------------------------------------------------------------

Purpose:
- estimate uncertainty for the difference:

Delta M = M_proposed - M_baseline

Potential metrics:
- Accuracy
- Balanced Accuracy
- Macro-F1
- MCC

Recommended:
- 2,000 or more bootstrap replicates for final reporting.

Output:
- observed metric difference;
- bootstrap mean difference;
- 95% CI.

------------------------------------------------------------
18.8 Statistical caution
------------------------------------------------------------

Do not say:
"Five CV folds are five fully independent observations."

CV folds are matched resampling units with overlapping training sets.

A careful manuscript statement is:

"Performance was summarized across matched cross-validation folds, while
paired non-parametric procedures were used for model comparisons. External
cross-dataset experiments were additionally repeated across multiple random
seeds."

============================================================
22. HOW TO INTERPRET FUTURE RESULTS
============================================================

When future numerical results are provided, analyze them in the following
order.

------------------------------------------------------------
22.1 First: verify experiment completeness
------------------------------------------------------------

Check:
- all intended models completed the same scenarios;
- paired comparisons use the same seeds/folds;
- no model has missing scenarios;
- class spaces match the protocol;
- no accidental overwrite or missing run occurred.

------------------------------------------------------------
22.2 Second: separate within-domain and cross-domain conclusions
------------------------------------------------------------

Do not combine them into one vague ranking.

A model can:
- be strongest within-domain;
- but weak under dataset shift.

Another model can:
- have slightly lower within-domain score;
- but smaller cross-dataset degradation.

These are scientifically different findings.

------------------------------------------------------------
22.3 Third: inspect transfer direction asymmetry
------------------------------------------------------------

Compare:

A->B vs B->A
A->C vs C->A
B->C vs C->B

A large difference indicates asymmetric domain shift.

Possible explanations:
- source sample diversity;
- target image difficulty;
- class balance;
- acquisition conditions;
- background complexity;
- label consistency.

Do not claim a cause without evidence; phrase causes as hypotheses or
inferences unless metadata support them.

------------------------------------------------------------
22.4 Fourth: analyze the headline six-class experiment
------------------------------------------------------------

A+C->B is the main six-class held-out target scenario.

Questions:
- Does multi-source training improve over A->B?
- Which classes improve?
- Which classes remain weak?
- Does the proposed model preserve its advantage?
- Does class-source confounding appear to affect source-specific classes?

Important:
A+C->B is not a standard pairwise task; it is multi-source external
evaluation.

------------------------------------------------------------
22.5 Fifth: examine per-class errors
------------------------------------------------------------

Use:
- confusion matrices;
- per_class_metrics.csv;
- predictions.csv.

Look for:
- Cercospora vs bacterial confusion;
- nutrient deficiency vs disease confusion;
- healthy false positives;
- powdery mildew transfer failures;
- curl-virus domain sensitivity.

Do not infer pathology explanations without visual/error evidence.

------------------------------------------------------------
22.6 Sixth: evaluate efficiency
------------------------------------------------------------

A lightweight model claim should not rely only on accuracy.

Compare:
- parameters;
- Macro-F1;
- MCC;
- runtime;
- GPU memory.

A strong result may be:
- similar Macro-F1 to a larger model;
- much fewer parameters;
- smaller memory use.

------------------------------------------------------------
22.7 Seventh: interpret statistics
------------------------------------------------------------

Preferred logic:

1. Check Friedman omnibus p-value.
2. If evidence of overall differences exists, inspect pairwise Wilcoxon tests.
3. Use Holm-adjusted p-values.
4. Report rank-biserial effect sizes.
5. For proposed vs strongest baseline, inspect McNemar and bootstrap CI.
6. Distinguish statistical significance from practical significance.

------------------------------------------------------------
22.8 Eighth: avoid overclaiming
------------------------------------------------------------

Do not claim:
- universal SOTA;
- pathogen-confirmed diagnosis;
- real-world deployment readiness;
- complete leakage elimination;
- external validation when describing pooled CV.

Use precise language:
- "cross-dataset evaluation"
- "held-out target dataset"
- "multi-source external evaluation"
- "exact duplicate audit"
- "public-dataset benchmark"
